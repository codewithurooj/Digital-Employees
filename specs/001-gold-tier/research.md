### 1. Odoo Integration Approach

#### Question
How to integrate with Odoo Community Edition v19+ for accounting data?

#### Findings

**Odoo External API Options**:

1. **XML-RPC (Built-in)** - Python's `xmlrpc.client`
   - Native to Odoo, no external library needed
   - Endpoints: `/xmlrpc/2/common` (auth), `/xmlrpc/2/object` (CRUD)
   - Authentication: database, username, password/API key
   - Supports all models: account.move, account.payment, res.partner

2. **JSON-RPC** - `httpx` or `requests`
   - Same capabilities as XML-RPC
   - Better for modern Python async patterns
   - Endpoint: `/jsonrpc`

3. **OdooRPC Library** (third-party)
   - Higher-level abstraction
   - Adds dependency, less control
   - Community maintained

#### Decision
**XML-RPC via `xmlrpc.client`** (Python built-in)

**Rationale**:
- Zero additional dependencies
- Odoo's documented, stable protocol
- Synchronous calls fit our watcher pattern
- Easy error handling with standard exceptions

#### Connection Pattern
```python
import xmlrpc.client

# Authentication
common = xmlrpc.client.ServerProxy(f'{url}/xmlrpc/2/common')
uid = common.authenticate(db, username, password, {})

# Model access
models = xmlrpc.client.ServerProxy(f'{url}/xmlrpc/2/object')
invoices = models.execute_kw(db, uid, password,
    'account.move', 'search_read',
    [[['move_type', '=', 'out_invoice'], ['state', '=', 'posted']]],
    {'fields': ['name', 'partner_id', 'amount_total', 'date']}
)
```

### 2. Social Media Platform APIs

#### Question
Best approach for Facebook, Instagram, and Twitter/X integration?

#### Findings

**Facebook/Instagram (Meta)**:

1. **Meta Graph API** (Official)
   - Requires Facebook Page (not personal profile)
   - Instagram Business Account linked to Page
   - Permissions: `pages_manage_posts`, `instagram_basic`, `instagram_content_publish`
   - Rate limits: 200 calls/hour per user
   - Supports: text posts, images, carousels, stories (limited)

2. **Playwright Browser Automation** (Fallback)
   - Works with personal profiles
   - Risk of account suspension if detected
   - Session management via cookies
   - No rate limits (natural user behavior)

**Twitter/X**:

1. **API v2** (Official)
   - Free tier: 1,500 tweets/month write, 10,000 reads/month
   - Basic tier ($100/mo): 50,000 tweets/month
   - OAuth 2.0 with PKCE
   - Supports: tweets, threads, media, polls

2. **Playwright Browser Automation** (Fallback)
   - Session-based authentication
   - Risk of account suspension
   - No API rate limits

#### Decision
**Official APIs (primary) + Playwright (fallback)**

**Rationale**:
- APIs provide reliability and metrics
- Playwright fallback for edge cases or API failures
- Constitution requires audit trails (APIs provide this)
- Browser automation is last resort, not default

#### Platform Comparison

| Feature | Facebook | Instagram | Twitter/X |
|---------|----------|-----------|-----------|
| Text posts | Yes (Graph API) | Yes (caption) | Yes (API v2) |
| Images | Yes | Yes (required) | Yes |
| Video | Yes | Yes (Reels via API) | Yes |
| Character limit | 63,206 | 2,200 caption | 280 (tweets) |
| Rate limit | 200/hr | 25 posts/day | 1,500/month (free) |
| Metrics | reach, engagement | likes, comments | impressions, engagement |

### 4. Ralph Wiggum Loop Pause/Resume

#### Question
How to implement pause/resume for mid-loop approvals?

#### Findings

**Current Implementation** (`src/utils/ralph_wiggum.py`):
- Loop state: loop_id, prompt, iterations, status, output_history
- Completion triggers: `TASK_COMPLETE` promise, file moved to Done
- Max iterations: 10, timeout: 300s per iteration
- No pause/resume support currently

**Pause/Resume Options**:

1. **State Serialization**
   - Save complete loop state to JSON file
   - Resume by loading state and continuing
   - Simple, reliable, vault-compatible

2. **Database State**
   - SQLite for complex state management
   - Adds dependency, complexity
   - Overkill for single-user system

3. **In-Memory Queue**
   - Orchestrator manages pause queue
   - Lost on restart
   - Not suitable for long approvals

#### Decision
**State Serialization to Vault**

**Rationale**:
- Consistent with vault-centric architecture
- Survives restarts
- Human-inspectable state files
- Simple implementation

#### Implementation Pattern
```python
class LoopState:
    def save(self, path: Path):
        """Serialize state to JSON for pause."""
        state = {
            'loop_id': self.loop_id,
            'prompt': self.prompt,
            'iterations': self.iterations,
            'output_history': self.output_history,
            'paused_at': datetime.now().isoformat(),
            'awaiting_approval': self.approval_request_id
        }
        path.write_text(json.dumps(state, indent=2))

    @classmethod
    def load(cls, path: Path) -> 'LoopState':
        """Deserialize state for resume."""
        state = json.loads(path.read_text())
        return cls(**state)
```

### 5. Multiple MCP Server Architecture

#### Question
How to manage multiple MCP servers with isolation?

#### Findings

**Current Architecture**:
- Single EmailMCP server in orchestrator
- Circuit breaker and rate limiter per server
- Health check pattern established

**Multi-Server Options**:

1. **Process Isolation**
   - Each MCP server as separate process
   - Communication via IPC/HTTP
   - Complex, overkill for local deployment

2. **Thread Isolation** (current pattern)
   - Each MCP server in dedicated thread
   - Shared memory for efficiency
   - Circuit breaker isolation sufficient

3. **Async Tasks**
   - All servers in single async event loop
   - Requires async rewrite of existing code
   - Better performance, more complexity

#### Decision
**Thread Isolation (extend current pattern)**

**Rationale**:
- Consistent with existing orchestrator
- Circuit breakers already provide fault isolation
- No architecture change needed
- Easy to debug and monitor

#### Server Configuration
```python
MCP_SERVERS = {
    'email': {
        'module': 'src.mcp_servers.email_mcp',
        'class': 'EmailMCP',
        'rate_limit': {'max_calls': 10, 'period': 3600},  # 10/hour
        'circuit_breaker': {'threshold': 5, 'timeout': 300}
    },
    'odoo': {
        'module': 'src.mcp_servers.odoo_mcp',
        'class': 'OdooMCP',
        'rate_limit': {'max_calls': 100, 'period': 3600},  # 100/hour
        'circuit_breaker': {'threshold': 3, 'timeout': 600}
    },
    'social': {
        'module': 'src.mcp_servers.social_mcp',
```

### 6. Audit Logging Format

#### Question
What logging format for 90-day audit retention?

#### Findings

**Current Implementation**:
- JSON logs in `Logs/YYYY-MM-DD.json`
- Fields: timestamp, action_type, actor, result
- Retained indefinitely (no cleanup)

**Audit Requirements** (from spec):
- ISO 8601 timestamp
- action_type, component, actor, target, parameters
- approval_status, result
- 90-day retention

**Options**:

1. **JSON Lines (JSONL)**
   - One JSON object per line
   - Easy append, hard to query
   - File-per-day rotation

2. **Structured JSON Array**
   - Single array per day
   - Requires read-modify-write
   - Better for querying

3. **SQLite**
   - Proper queries, indexes
   - Adds complexity
   - Not vault-native

#### Decision
**JSON Lines (JSONL) with daily rotation**

**Rationale**:
- Append-only is safe for concurrent writes
- Human-readable, git-friendly
- Easy to implement retention (delete old files)
- jq/grep for ad-hoc queries

#### Log Entry Format
```json
{
  "timestamp": "2026-01-25T14:30:00.000Z",
  "action_type": "social_post_publish",
  "component": "social_mcp",
```
